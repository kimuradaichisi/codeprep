# 新規プロジェクト要件定義書：ドキュメント関係性解析・可視化インデクサー

本ドキュメントは、CodePrep のドキュメント群（設計書、仕様書など）連携機能を拡張するにあたり、独立した新規プロジェクトとして立ち上げる「ドキュメント関係性解析・可視化インデクサー（仮称: DocIndexer / DocPrep）」の検討プロセス、ユースケース、およびシステム要件をまとめたものです。

---

## 1. 開発背景とビジョン

### 1-1. 現状の課題（CodePrep における課題）
- LLMへのコンテキスト注入時、ソースコードだけではなく「設計書や仕様書などの関連ドキュメント」も併せて渡すことが文脈理解に極めて重要である。
- しかし、人間は面倒くさがってドキュメント間に明示的な `[リンク](path)` を書かなかったり、古いリンクが放置されたりする。
- ドキュメント全体のテキストを丸ごと送るとトークン容量を圧迫するため、関連部分（周辺行）のみをスマートに切り出す必要がある。
- これらをすべて CodePrep 単体で処理しようとすると、ドキュメント解析のオーバーヘッドや外部ライブラリ（RAG/DB）の肥大化により、CodePrep 自体の軽量さ・ポータビリティ（単一 `.exe` 動作）が損なわれる。

### 1-2. 解決策：新規独立プロジェクトの立ち上げ
- ドキュメント群の「解析」「インデックス化」「関係性グラフ可視化」を行う**独立した別プロジェクト**を立ち上げる。
- 解析した関係性は、最終的に軽量なインデックス定義ファイル（`codeprep.map.json`）としてエクスポートする。
- CodePrep はこのエクスポートされた JSON を単に読み込んで動作することで、お互いのポータビリティと疎結合を維持する。

---

## 2. 検討プロセスと機能アプローチ

リンクのない乱雑なドキュメント群から関係性を安全かつ賢く紐付けるため、以下の3つのアプローチを検討しました。

### ① インプリシット・リンク（本文自動キーワードマッチ）
- **仕組み**: プロジェクト内の全ドキュメント名（例: `認証仕様書.md` などの固有名詞）をインデックス化し、他のドキュメント本文からその単語の出現を自動スキャンして「隠れたリンク」として紐付ける。
- **効果**: 人間がリンクを貼る手間を完全に排除する。

### ② Git 履歴による「同時変更（Co-change）」分析
- **仕組み**: 過去の Git コミット履歴を解析し、「いつもセットで同時にコミットされているファイル群（例: `決済仕様.md` と `返金仕様.md`）」を統計的に紐付ける。
- **効果**: テキスト本文に依存せず、開発の実績から実用的な関係性を抽出できる。

### ③ グラフデータベース ✕ 可視化 UI
- **仕組み**: ドキュメント間のつながりを有向グラフとしてデータ構造化し、`D3.js` などを利用してインタラクティブなネットワークグラフとしてビジュアル表示する。
- **効果**: 孤立した（誰も参照していない）古い仕様書の発見や、システム全体の設計ドキュメントの網羅性を視覚的に把握できる。

## 3. システム要件（要件定義）

新規プロジェクトが満たすべき機能要件および非機能要件です。

### 3-1. 機能要件

#### A. ドキュメント解析・インデックス生成機能
- **Markdown / プレーンテキスト解析**:
  - ファイル内の見出し（Heading）、リンク（Markdown 相対パス、Wikiリンク）、およびフロントマター（Tags）の自動抽出。
- **キーワード逆引きスキャン**:
  - 別のファイル名が本文中に登場している箇所を自動検出し、暗黙 of 依存関係として登録。
- **Git 履歴スキャナー**:
  - 対象リポジトリのコミットログから、同時修正頻度の高いファイルペアの重み付け抽出。
- **インデックス書き出し**:
  - 解析結果を `codeprep.map.json` 等の静的 JSON マップとして保存。

#### B. グラフ可視化ダッシュボード (UI)
- **ネットワークグラフ描画**:
  - ドキュメントを「ノード」、関係性を「エッジ（矢印）」としたインタラクティブな 2D/3D グラフ表示。
- **フィルタリング・検索**:
  - 特定のキーワードや機能グループでグラフのノードを絞り込み表示する機能。
- **手動紐付けエディタ**:
  - UI上でノード同士をドラッグ＆ドロップして関係性（エッジ）を追加・編集・削除し、JSON マップへ逆反映する機能。

### 3-2. 非機能要件
- **ポータビリティ**: 開発環境で手軽に動かせるよう、軽量な DB（SQLite や DuckDB）またはローカルメモリ内でのデータ保持を優先する。
- **RAG/ベクトルの活用**: 大規模プロジェクト向けに、ローカル完結型の軽量ベクトルデータベース（`sqlite-vss` 等）を用いたセマンティック検索機能のプラグイン化対応。
- **CodePrep 互換性**: CodePrep がパースしやすいシンプルで平滑化された JSON フォーマットでのエクスポート機能。
